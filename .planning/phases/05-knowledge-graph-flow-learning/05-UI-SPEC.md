# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase 5 UI surface:** the **Knowledge Graph browse** section — a small set of new authenticated, read-only pages under `app/(dashboard)/graph/` that let the user query and tabularly browse the discovered knowledge graph (KG-02, D-05). It comprises three list views plus their drill-in detail surfaces:

1. **Pages** (`/graph` index → "Pages" tab/section) — every discovered `:Page`, with freshness, and drill-in to a page's elements, forms, and outbound navigation edges.
2. **Flows** (`/graph/flows`) — derived business journeys (`GET /flows`), each with a **risk-score badge** (0–100 → `--status-*` tiers) and a category name, drill-in to the ordered step sequence.
3. **Element Repository** (`/graph/elements`) — every discovered `:Element` with its prioritized locator chain and locator history (KG-05 half), drill-in to per-element detail.

A **coverage** figure (`GET /coverage`, % vs the hand-labeled SauceDemo ground truth, D-08/QUAL-01) is surfaced honestly as a header stat on the Pages view. Plus one small shared addition: a **"Knowledge graph" sidebar nav item** and the activation of the **"View knowledge graph"** inline link that the Phase-4 terminal banner already references.

**Design system is LOCKED by 01-UI-SPEC.md, extended by 04-UI-SPEC.md, and confirmed in `apps/web/components.json` + the vendored `apps/web/components/ui/` on disk — it is REUSED verbatim here, not redefined.** This document specifies only what is NEW for the KG browse: the page set, each view's columns/cards, how relationships render as drill-in links, the risk-score badge mapping, the honest coverage display, and the empty/loading/error states — all in the established voice. Every spacing/typography/color value below REFERENCES the Phase-1 tokens.

**AUTO-mode note:** No design-system questions were asked — 01-UI-SPEC (approved 2026-06-12), 04-UI-SPEC (approved 2026-06-15), 05-CONTEXT D-05/D-06, and 05-RESEARCH (the read-API response shapes, node/edge schema, risk-tier thresholds, coverage shape, the "empty graph without a live exploration" reality) already answered every contract question. Sources are cited per section; anything not sourced is a documented default.

## Color

Inherited verbatim from 01-UI-SPEC dark-first zinc palette, with the **same `--status-*` token reuse** that 04-UI-SPEC established for run states — now mapped onto the deterministic 0–100 risk score. No new palette, no new tokens.

| Role | Value | Usage on the KG browse |
|------|-------|------------------------|
| Dominant (60%) | #09090B (zinc-950) | Page background / `<main>` surface |
| Secondary (30%) | #18181B (zinc-900), borders #27272A (zinc-800) | Table surface + header row, stat cards, drill-in detail panel cards, tab bar |
| Accent (10%) | #3B82F6 (blue-500) | See reserved list below |
| Destructive | #EF4444 (red-500) | not used as an action color this section (these pages are read-only — no destructive action exists here); the red HUE appears ONLY via the shared `--status-fail` token inside a high-risk badge, never as a button |

Text: primary #FAFAFA (zinc-50); muted #A1A1AA (zinc-400) for table column headers, timestamps, fingerprint sub-labels, the coverage context line, stale-row de-emphasis, and breadcrumbs.

**Accent reserved for:** the active "Knowledge graph" sidebar nav indicator; focus rings on all interactive elements (2px accent ring, inherited interaction default); the **drill-in links** (every relationship rendered as a link — page→element, page→form, page→navigates-to, flow→steps, element→detail — is an inline accent link, the primary interaction of these pages); the active tab/segment indicator on the `/graph` view switcher; and the "View knowledge graph" / "Back to …" inline links. Never for borders, table rows at rest, icons-at-rest, risk badges, the coverage stat, or column headers (those use zinc neutrals + the `--status-*` tokens).

**Risk-score → status-token mapping (REUSE the existing `--status-*` tokens — do NOT introduce new colors).** The deterministic 0–100 risk score (D-04) maps onto the same three tiers Phase 1 pre-reserved and Phase 4 reused, with the thresholds proposed in 05-RESEARCH Pattern 3 (tunable):

| Risk tier | Range | Token | Resolved | Badge treatment |
|-----------|-------|-------|----------|-----------------|
| High | `>= 67` | `--status-fail` (#EF4444 red-500) | red | `badge` with a red dot + the numeric score (mono) + "High"; red used because a high-risk flow is the thing the user must act on, consistent with fail semantics |
| Medium | `34..66` | `--status-quarantine` (#F59E0B amber-500) | amber | `badge` with an amber dot + the numeric score + "Medium" |
| Low | `< 34` | `--status-pass` (#22C55E green-500) | green | `badge` with a green dot + the numeric score + "Low" |
| Unscored | n/a (no flows / no key for categorization) | `--status-neutral` (#A1A1AA zinc-400) | muted | `badge` muted, label "—" — never a fabricated number |

**Freshness color cue:** a node's `last_verified` recency is shown as a caption timestamp, NOT a color. A node flagged stale (research: marked by `last_verified` age, never deleted this phase) renders its row text in muted (zinc-400) with a `Clock`/`History` lucide icon + a `tooltip` "Last verified {timestamp} — not seen in the most recent exploration." Stale is a muted de-emphasis, never red (it is not a failure).

Rationale for reuse: green/amber/red/neutral were pre-reserved in Phase 1 as pass/quarantine/fail/neutral semantics precisely so derived scores inherit them with zero new color decisions; Phase 4 already bound them to run states, and the risk badge follows the same hue logic (high risk = the fail hue, the actionable signal).

## Screen & State Inventory (Phase 5)

Supplementary contract for planner/executor. Every view ships every listed state (web e2e per 05-RESEARCH `graph-browse.spec.ts`, against a mocked API, should touch them).

### Layout shell (all KG views, inside the existing `<main className="p-6">`, single column, desktop-only dev tool)

1. **Page header block** (`flex items-center justify-between`, `lg` gap below): Left — Heading "Knowledge graph". Right — the **coverage stat** rendered as a compact secondary-surface stat card (`md` padding): caption "Coverage" on top, Label-weight value below (mono `83.0%`, or "Not yet measured" muted), and the caption context line beneath. The coverage card is shown on the **Pages** view; on Flows/Element views it is omitted (it is a page-coverage figure).
2. **View switcher** (`md` gap below header): a segmented tab control ("Pages" · "Flows" · "Element repository") built as plain accent-underlined links/segments over existing tokens (NOT a new tabs block — composition is sufficient; the active segment carries the accent indicator). Each segment maps to a route (`/graph`, `/graph/flows`, `/graph/elements`) so views are deep-linkable.
3. **Content region** (`xl` gap below the switcher): the active view's table or detail panel.

### 1. Pages view (`/graph`)

- **Table** (secondary surface) columns: **Page** (title, Body, the row is the drill-in link to page detail) · **URL** (mono, truncated with `title`) · **Fingerprint** (mono, truncated to ~8 chars + `tooltip` full value) · **Elements** ("{n} elements" accent drill-in link → detail Elements section) · **First seen** (caption mono timestamp) · **Last verified** (caption mono timestamp; stale rows muted with the `Clock` icon + tooltip).
- **States:** loading (skeleton rows + skeleton coverage stat) · **empty — no graph** (the "No knowledge graph yet" empty state centered in the table region, coverage shows "Not yet measured") · populated · error (inline + Retry) · graph-profile-down (inline Neo4j message + Retry).

### 2. Flows view (`/graph/flows`)

- **Table** (secondary surface) columns: **Flow** (name — LLM category-derived or deterministic fallback — Body; the row drills into flow detail) · **Category** (Body, e.g. "Authentication", "Checkout"; "—" muted when unnamed) · **Risk** (the risk badge: colored dot + mono score + tier word) · **Steps** ("{n} steps" accent drill-in link → flow detail Steps section) · **First seen** (caption mono). Default sort: **risk descending** (highest-risk flows first — the actionable ordering).
- **States:** loading (skeleton rows) · **empty — no flows** ("No flows derived yet") · empty — no graph (the shared "No knowledge graph yet" state) · populated · error (inline + Retry).

### 3. Element repository view (`/graph/elements`)

- **Table** (secondary surface) columns: **Element** (label, Body — the drill-in link to element detail) · **Role** (Body, e.g. button/link/textbox) · **Page** (host page title, accent link → that page's detail) · **Locator (top priority)** (mono — the highest-priority entry of the chain, e.g. the `data-testid` value; truncated with tooltip) · **Last verified** (caption mono; stale muted).
- **States:** loading (skeleton) · **empty — no elements** ("No elements captured yet") · empty — no graph (shared state) · populated · error (inline + Retry).

### 4. Drill-in detail panels (rendered as routed detail pages or side `sheet`s — planner's discretion; the contract is the content + sections below)

- **Page detail** (`/graph/pages/[fingerprint]` or sheet): breadcrumb caption; the page title (Heading) + url + fingerprint + first_seen/last_verified (mono captions). Sections (Label headings, `lg` gaps, each a card or separated block): **Elements** (a compact table/list of the page's `:Element`s, each linking to element detail) · **Forms** (the page's `:Form`s with their validation rules summarized) · **Navigates to** (a list of outbound `NavigatesTo` edges — each "Navigates to {target title}" an accent link to that page's detail; the edge's `via` control label shown as a mono caption). Optional: the page's latest screenshot in the inherited 16:10 frame if a `screenshot_path` is available.
- **Flow detail** (`/graph/flows/[id]` or sheet): breadcrumb; flow name (Heading) + category + the risk badge. Sections: **Steps** — the ordered page sequence (`:STEP {order}`) as a numbered list, each step linking to that page's detail · **Risk breakdown** — the deterministic signals behind the score (has-destructive yes/no, state-changing edge count, auth-gated step count, form count, path length) shown as a small caption/value list so the score is auditable (D-04 "explainable").
- **Element detail** (`/graph/elements/[key]` or sheet): breadcrumb; element label (Heading) + role + host page (link). Sections: **Locator chain** — the prioritized chain rendered as an ordered list, each row a mono priority label (`data-testid` / `aria-label` / `role` / `text` / `xpath`) → its mono value; the currently-active/winning locator visually flagged · **Locator history** — prior locator values with their `first_seen`/`last_verified` (mono captions) so the user sees how a locator changed over runs (KG-05).
- **States (every detail):** loading (skeleton) · populated · not-found ("No {page/flow/element} found." + breadcrumb link back to the list) · error (inline + Retry).

### Shared addition

- **"Knowledge graph" sidebar nav item** — appended to `NAV_ITEMS` in `components/app-sidebar.tsx` following the documented flat-list `{icon, label, href}` contract; `href: "/graph"`, active via `pathname.startsWith("/graph")`. Positioned after "Explorations" (the natural reading order: explore → graph).

### Data fetching & boundary

- All views fetch via `@tanstack/react-query` against the read endpoints (`GET /api/graph` or `/api/pages`, `GET /api/flows`, `GET /api/elements`, `GET /api/coverage`) through the existing same-origin `/api/:path*` rewrite, so the httpOnly auth cookie rides automatically (consistent with `lib/api/client.ts`). Responses are validated at the boundary with `zod` schemas in `lib/api/kg.ts` that mirror the Pydantic response models (05-RESEARCH). No mutations, no optimistic updates, no SSE — these are static read snapshots; a manual "Retry" re-runs the query (no auto-polling — the graph is not live-changing on this surface).
- Endpoints are `Depends(get_current_user)`-gated server-side; an unauthenticated request returns 401 and the existing client wrapper redirects to `/login` (no per-page auth handling).

### Accessibility

- **Tables:** real `<table>` semantics via shadcn `table` (column `<th scope="col">`, caption where helpful). Sortable columns (Flows by risk) expose `aria-sort`.
- **Risk badge:** the tier is NEVER color-only — every badge carries the tier WORD ("High"/"Medium"/"Low") and the numeric score as text alongside the colored dot (WCAG 1.4.1). The dot is `aria-hidden`; the text carries the meaning. A `tooltip` on the badge gives the full breakdown summary.
- **Stale rows:** the muted color is reinforced by the `Clock`/`History` icon (`aria-hidden`) AND a real text tooltip — staleness is not conveyed by color alone.
- **Drill-in links:** every relationship is a real `<a>`/`<Link>` reachable by Tab with the 2px accent focus ring (inherited); no interaction is hover-only. Truncated mono values (URL, fingerprint, locator) carry a `title` (or tooltip) so the full value is available without hover-only reliance for AT.
- **Coverage stat:** the value + context are real text (e.g. `aria-label="Coverage: 83.0 percent, 6 of 7 ground-truth pages discovered"`); "Not yet measured" is real text, never an empty or fabricated value.
- **Empty/error regions:** rendered as text with a reachable action link/button (Go to targets / Retry); never an icon-only or color-only state.
- **Breadcrumbs:** an `aria-label="Breadcrumb"` nav with the current page marked `aria-current="page"`.
- **Motion:** no streaming/animation on this surface (static reads); nothing to gate on `prefers-reduced-motion` beyond inherited component defaults.

### Interaction defaults (inherited from 01-UI-SPEC, applied here)

- Focus rings: 2px accent ring on all interactive elements (links, tabs, Retry).
- Sentence case throughout; verbs-first on the only buttons present (Retry).
- Errors render inline in the content region, never as toasts; there are no success toasts (no mutations).
