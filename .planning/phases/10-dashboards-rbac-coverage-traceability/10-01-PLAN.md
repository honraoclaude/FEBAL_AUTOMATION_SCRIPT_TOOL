<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → API (cookie auth) | every request crosses here; the role decision must be server-side |
| authenticated user → role-assignment endpoint | a user must not be able to elevate their own (or escalate) privilege |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Elevation of Privilege | POST /api/users/{id}/role | mitigate | router-level `require_role("admin")`; a non-admin gets 403 — proven in test_role_assign |
| T-10-02 | Elevation of Privilege | admin self-demote / lockout | mitigate | the current admin's own row cannot be changed (400) — guards against locking out the only admin |
| T-10-03 | Spoofing / Tampering | forged role in token | mitigate | role is NOT in the JWT — read off the users row each request; nothing to forge (A1) |
| T-10-04 | Elevation of Privilege | stale-role window after a role change | mitigate | role read off the row per request → a change takes effect on the next request, no token reissue |
| T-10-05 | Tampering | migration 0010 breaks existing rows | mitigate | server_default='admin' on the column; reversible up/down/up gate (test_migration_0010) |
| T-10-06 | Information Disclosure | leaking secrets in role-change logs | accept | role changes log id/email/role only — no secret/token field is touched (existing redaction processor) |
</threat_model>
