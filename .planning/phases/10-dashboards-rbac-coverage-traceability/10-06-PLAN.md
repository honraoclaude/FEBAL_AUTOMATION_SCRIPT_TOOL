<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → coverage/traceability/search/users API | role-scoped access; the API is the boundary, the UI mirrors |
| admin role-change | a user must not self-demote/escalate; the UI mirrors the server guard |
| search query input | untrusted; rendered highlight must not enable script injection |
| traceability/search drill links | must link to existing pages (which own their auth + 404) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-27 | Elevation of Privilege | admin users screen reachable by a non-admin | mitigate | the API gates /api/users (Plan 01); the nav hides Users; a direct URL → 403 → the no-access state, never the data |
| T-10-28 | Elevation of Privilege | self-demote / lockout via the UI | mitigate | the self-row control is disabled with the guard caption; the server 400 is the real boundary |
| T-10-29 | Tampering / XSS | rendering the ES highlight fragment | mitigate | render the server highlight as safe emphasized text (no unsanitized HTML injection); q is a query value, not markup |
| T-10-30 | Information Disclosure | a fabricated chain link / search hit / coverage number | mitigate | every node/hit/metric renders strictly from the server payload; missing → honest gap; ES-down → honest 503, never a fake empty list |
| T-10-31 | Information Disclosure | conflating the two coverage metrics | mitigate | separate cards each with its definition (Pitfall 5) |
</threat_model>
