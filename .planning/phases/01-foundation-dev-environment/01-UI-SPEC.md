# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase 1 UI surface:** login page (email/password), authenticated app shell (sidebar nav that later phases extend — graph browser, review queues, dashboards), and the target-application registry (table + create/edit dialog with write-only masked credentials and exploration rules). UI-bearing requirements: PLAT-01, PLAT-03. Aesthetic: dark-mode-first developer/QA tool.

**AUTO-mode note:** All decisions below were made without user Q&A per the user's standing delegation (CONTEXT.md D-01). Sources are cited per section; anything not sourced from CONTEXT/RESEARCH/STACK is a documented default.

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (CLI latest) — locked by STACK.md; `npx shadcn@latest init` runs inside the web-shell scaffold plan (app does not exist yet, so init is deferred to execution, with the preset values above pinned here) |
| Preset | `new-york` style, base color `zinc`, CSS variables `true`, Tailwind 4 CSS-first `@theme` (no tailwind.config.js — per RESEARCH State of the Art) |
| Component library | radix-based shadcn defaults |
| Icon library | lucide-react 1.x (STACK.md lock; shadcn default) |
| Font | Geist Sans (UI) + Geist Mono (URLs, credentials masks, code/log output) — both ship with `create-next-app@16`, zero extra dependency |

**Theme mode decision (default, documented):** Dark theme only in Phase 1. The `<html>` element gets the dark class permanently; no theme toggle. Tokens are defined as CSS variables so a light theme can be added in a later phase without component changes. Rationale: developer-tool product, solo user, smallest surface.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding |
| sm | 8px | Gaps inside form rows, table cell vertical padding, button icon gaps |
| md | 16px | Default gap between form fields, card padding, dialog content padding |
| lg | 24px | Page content padding, gap between page header and content, dialog section breaks |
| xl | 32px | Gap between major page regions (header / table / footer) |
| 2xl | 48px | Login card top offset grouping, major section breaks |
| 3xl | 64px | Page-level vertical rhythm on the login screen |

Exceptions: none. (Desktop-only developer tool; default shadcn control heights — `h-9` 36px inputs/buttons — are multiples of 4 and remain within contract.)

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 14px | 600 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

- Exactly 4 sizes total when including the 12px caption noted below; exactly 2 weights (400, 600). No other sizes or weights may appear.
- **Caption/meta (12px / 400 / 1.5):** table column headers, helper text under form fields, timestamp metadata. This is the 4th declared size.
- **Heading (20px)** = page titles ("Target Applications") and dialog titles. **Display (28px)** = login screen product heading only.
- **Monospace rule:** Geist Mono at the surrounding role's size (no new sizes) for: target base URLs, origin allowlist entries, credential mask placeholders, and any future code/log output. This anticipates later phases (logs, locators, Gherkin) per the roadmap.

## Color

Dark-first zinc palette (matches shadcn `zinc` base). Values are the resolved dark-theme hex values; implemented as CSS variables via shadcn init.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #09090B (zinc-950) | App background, page surfaces |
| Secondary (30%) | #18181B (zinc-900), borders #27272A (zinc-800) | Sidebar, cards, dialogs, table header row, input backgrounds |
| Accent (10%) | #3B82F6 (blue-500) | See reserved list below |
| Destructive | #EF4444 (red-500) | Destructive actions and validation errors only |

Text: primary #FAFAFA (zinc-50), muted #A1A1AA (zinc-400). Muted text is for captions, helper text, and inactive-target rows — never for primary content.

Accent reserved for: primary action buttons ("Log in", "Register target", "Save changes"), focus rings, the active sidebar nav item indicator, and inline links. Never for borders, icons-at-rest, table rows, badges, or secondary buttons (those use zinc neutrals).

**Accent hue decision (default, documented):** blue, not green/violet — green and red are pre-reserved as pass/fail status semantics for later phases. Define these status tokens in `@theme` now so Phase 5+ inherits them; Phase 1 uses only the two listed below:

| Future status token | Value | Phase 1 usage |
|---------------------|-------|---------------|
| `--status-pass` | #22C55E (green-500) | "Active" target badge (dot + label) |
| `--status-fail` | #EF4444 (shares destructive) | none in Phase 1 |
| `--status-quarantine` | #F59E0B (amber-500) | Sandbox badge on targets table (amber-outline — warns destructive actions are permitted); heal quarantine reuses it in Phase 8 |
| `--status-neutral` | #A1A1AA (zinc-400) | "Inactive" (soft-deleted) target badge |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (login) | "Log in" |
| Primary CTA (targets page) | "Register target" |
| Dialog submit CTA (create) | "Register target" |
| Dialog submit CTA (edit) | "Save changes" |
| Empty state heading | "No target applications yet" |
| Empty state body | "Register a target application to give the platform something to explore." → button: "Register target" |
| Error state (login failure) | "Invalid email or password." (uniform message for unknown user vs wrong password — security posture per RESEARCH V2; no recovery hint beyond retry) |
| Error state (request failure) | "Couldn't save the target. Try again — if it keeps failing, check that the API container is healthy (`docker compose ps`)." |
| Error state (field validation) | Inline under field, 12px destructive text: "Name is required" / "Enter a valid URL (including http:// or https://)" |
| Destructive confirmation | Deactivate target: "Deactivate {name}? The platform will stop running against this target. Its history is kept and you can reactivate it later." Confirm button (destructive variant): "Deactivate" / Dismiss: "Keep target" |
| Credential masking (edit dialog) | Helper text under credential fields: "Stored encrypted and never shown. Enter new values to replace them." Placeholder in both fields: "••••••••" |

Voice rules: sentence case everywhere (buttons, headings, labels — "Register target", not "Register Target"); verbs first on buttons; no exclamation marks; error copy states the problem then the next step.

## Screen & State Inventory (Phase 1)

Supplementary contract for planner/executor — every screen ships every listed state (Playwright e2e per D-02 should touch them).

### 1. Login (`/login`)
- Centered card (max-w-sm) on dominant background; Display heading (product name), email + password fields, full-width accent "Log in" button.
- States: default · submitting (button disabled, spinner, label unchanged) · error (uniform error message above the button, fields retain email value, password cleared).
- No signup link, no "forgot password" (D-03: env-seeded admin only). Footer caption (12px muted): "Admin account is provisioned from environment configuration."

### 2. App shell (authenticated layout)
- shadcn sidebar (secondary surface, 256px) + main content area (dominant surface, `lg` padding).
- Sidebar contents Phase 1: product wordmark (top), nav item "Targets" (active, accent indicator), and a muted "More coming" disabled group is NOT shown — only render nav items that exist. Bottom: user email (muted, 12px) + "Log out" action.
- Later phases append nav items (Explorations, Graph, Scenarios, Executions, Dashboards) — the sidebar component contract is: flat list of `{icon, label, href}`, active state derived from route.
- Log out: immediate action, no confirmation (non-destructive; D-04 client-side logout), redirects to `/login`.

### 3. Target registry (`/targets`)
- Page header: Heading "Target Applications" left, accent "Register target" button right.
- Table (secondary surface) columns: Name · Base URL (mono) · Sandbox (badge: "Sandbox" amber-outline when true, blank when false) · Status (Active green-dot badge / Inactive muted badge) · row-actions dropdown (Edit, Deactivate/Reactivate) — icon-only trigger carries `aria-label="Actions for {name}"`.
- States: loading (skeleton rows) · empty (empty-state copy above, centered in table region) · populated · row inactive (entire row text muted; Reactivate replaces Deactivate in the menu).

### 4. Register/Edit target dialog
- shadcn Dialog, `md` content padding, single column, sections separated by `lg` gaps with Label-weight section headings: "Target" (name, base URL) · "Credentials" (username, password — password input type, masked helper copy on edit) · "Exploration rules" (origin allowlist as textarea one-origin-per-line with helper "Defaults to the base URL's origin", sandbox Switch with caption "Allows destructive actions during exploration — only enable for restorable targets", collapsible "Budget overrides" group of 4 optional numeric fields: max steps, max depth, wall-clock seconds, token budget).
- States: create default · edit prefilled (credentials masked, never prefilled) · submitting · field-level validation errors · request error (inline alert at top of dialog body).
- Cancel discards silently (no dirty-state confirmation in Phase 1 — documented default to keep scope thin).

### Interaction defaults
- Focus rings: 2px accent ring on all interactive elements (keyboard visible).
- Toasts (sonner, bottom-right): success only — "Target registered" / "Target updated" / "Target deactivated". Errors render inline, never as toasts.
- All mutations via TanStack Query mutations with list invalidation; no optimistic updates in Phase 1.
