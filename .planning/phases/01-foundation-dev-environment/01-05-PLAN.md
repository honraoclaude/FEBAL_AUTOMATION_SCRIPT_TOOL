## Phase Goal

**As a** platform operator, **I want to** stand up the platform skeleton locally with one command, log in, and securely register a target application, **so that** the platform has a target to explore in later phases.

(This plan delivers "securely register a target application" via the API — PLAT-01's API half and all of PLAT-07. After it, a user can register, list, edit, deactivate, and reactivate targets with curl/httpx, with credentials encrypted and unleakable.)

Purpose: PLAT-01 (API), PLAT-07. The targets schema (sandbox flag, budget overrides, allowlist) is the Phase 4 Explorer's input contract.
Output: /api/targets CRUD live; test_targets.py and test_credential_security.py green.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/targets | untrusted input (URLs, budgets, credentials) crosses here |
| API ↔ Postgres | credential plaintext must never cross — only Fernet ciphertext is stored |
| API → logs | registration payloads flow through logging middleware |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-01-17 | Information Disclosure | credentials in API responses | mitigate | TargetResponse structurally credential-free (whitelist by omission); test_credentials_never_in_response asserts raw response text |
| T-01-18 | Information Disclosure | credentials in logs | mitigate | structlog redact_sensitive (plan 01-02) + test_logs_contain_no_plaintext capturing real container logs |
