# Phase 6 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase 6 UI surface:** the **Scenario Review Queue** — a small set of new authenticated pages under `app/(dashboard)/scenarios/` that let the reviewer triage LLM-generated BDD scenarios before they feed Playwright codegen (GEN-02). It is the **review step** that sits between generation and codegen; **generation and codegen are triggered elsewhere (from a flow) and are OUT of this UI's scope.** It comprises:

1. **Review queue list** (`/scenarios`) — every generated scenario as a row with a **status badge** (draft / approved / rejected), the **source flow + its risk tier**, the feature/scenario title, and a draft/approved/rejected/all filter. Default view: drafts (the queue of work).
2. **Scenario detail / review** (`/scenarios/[id]`) — the Gherkin in an **editable code area**, the **per-`Then` KG-assertion resolution indicators** (the honest output of the no-vacuous gate — D-03), and the **approve / reject / save-edit** actions.

The defining interaction is **edit-in-place** (D-02): the reviewer edits the Gherkin, saves, the syntax + no-vacuous gates **re-run server-side**, and inline pass/fail results surface with the offending `Then` highlighted. **Approve is enabled only when both gates pass.** Gate results are shown **honestly — a green indicator means the `Then` actually resolved to a Neo4j-recorded outcome; it is never fabricated.**

**Design system is LOCKED by 01-UI-SPEC.md, extended by 04-UI-SPEC and 05-UI-SPEC, and confirmed in `apps/web/components.json` + the vendored `apps/web/components/ui/` on disk — it is REUSED verbatim here, not redefined.** This document specifies only what is NEW for the review queue: the page set, the list columns + filter, the detail/review layout, the editable Gherkin area, the per-`Then` gate indicators, the edit→re-validate→approve flow, the empty/loading/error states, and the sidebar nav entry — all in the established voice. Every spacing/typography/color value below REFERENCES the Phase-1 tokens.

**AUTO-mode note:** No design-system questions were asked — 01-UI-SPEC (approved 2026-06-12), 05-UI-SPEC (approved 2026-06-19), 06-CONTEXT (D-01..D-04), and 06-RESEARCH (the `scenarios` schema, `then_refs` shape, review-router endpoints, status lifecycle, gate semantics) already answered every contract question. Sources are cited per section; anything not sourced is a documented default.

## Color

Inherited verbatim from 01-UI-SPEC dark-first zinc palette, reusing the **same `--status-*` tokens** Phase 1 pre-reserved and Phases 4/5 bound — now applied to (a) the **scenario status** lifecycle and (b) the **per-`Then` gate resolution** indicators, plus the **flow risk tier** carried straight from 05-UI-SPEC. No new palette, no new tokens.

| Role | Value | Usage on the review queue |
|------|-------|---------------------------|
| Dominant (60%) | #09090B (zinc-950) | Page background / `<main>` surface; the Gherkin `<textarea>` field background |
| Secondary (30%) | #18181B (zinc-900), borders #27272A (zinc-800) | Table surface + header row, filter bar, detail-panel cards, gate-results card, the `<textarea>` border |
| Accent (10%) | #3B82F6 (blue-500) | See reserved list below |
| Destructive | #EF4444 (red-500) | The **Reject** action (destructive variant button) and inline gate-failure messaging text; also shares the hue with `--status-fail` used for the "Vacuous"/rejected indicators |

Text: primary #FAFAFA (zinc-50); muted #A1A1AA (zinc-400) for table column headers, timestamps, the "edited" sub-label, rejected-row de-emphasis, gate-context lines, and breadcrumbs.

**Accent reserved for:** the active "Scenarios" sidebar nav indicator; focus rings on all interactive elements (2px accent ring, inherited) including the Gherkin `<textarea>`; the **primary action button "Approve scenario"** (the one accent CTA on the detail view); the active filter segment indicator on the queue filter bar; the scenario-title **drill-in link** from the list row to detail; and the breadcrumb "Back to review queue" link. Never for borders, table rows at rest, icons-at-rest, status badges, risk badges, gate indicators, or secondary buttons ("Save edits", "Cancel" — those use zinc neutrals; "Reject" uses the destructive variant).

**Scenario status → status-token mapping (REUSE the existing `--status-*` tokens — do NOT introduce new colors):**

| Status | Token | Resolved | Badge treatment |
|--------|-------|----------|-----------------|
| Draft | `--status-quarantine` (#F59E0B amber-500) | amber | `badge` amber dot + "Draft" — awaiting review (the queue's working state) |
| Approved | `--status-pass` (#22C55E green-500) | green | `badge` green dot + "Approved" — passed gates + reviewer accepted; feeds codegen |
| Rejected | `--status-neutral` (#A1A1AA zinc-400) | muted | `badge` muted dot + "Rejected"; the row text is also muted (de-emphasized, not red — a rejected scenario is set-aside, not a failure to alarm on) |

**Per-`Then` gate resolution → status-token mapping (the HONEST no-vacuous gate display — D-03):**

| Gate outcome | Token | Resolved | Indicator treatment |
|--------------|-------|----------|---------------------|
| Resolved (the `Then` maps to an existing Neo4j node/edge) | `--status-pass` (#22C55E green-500) | green | green `CheckCircle2` icon + "Resolved" (Label) + the resolved `kg_ref` shown as a mono caption (e.g. `Updates → Cart`). Green appears ONLY when the backend confirms resolution — never optimistically. |
| Vacuous (no ref, unresolved ref, unknown kind, or no `Then`) | `--status-fail` (#EF4444 red-500) | red | red `XCircle` icon + "Vacuous" (Label) + a mono caption naming why (`no graph-backed outcome` / the unresolved ref). The offending `Then` row is highlighted (left red border + muted-red tint). |
| Not yet checked (edit pending save) | `--status-neutral` (#A1A1AA zinc-400) | muted | muted `Circle`/`Clock` icon + "Pending re-check" — shown on edited-but-unsaved `Then`s; never green until the gate actually runs. |

**Flow risk tier (carried verbatim from 05-UI-SPEC — same `--status-*` tiers, same thresholds):** each scenario's source flow shows its risk badge so reviewers prioritize high-risk flows. High `>=67` → `--status-fail` red dot + mono score + "High"; Medium `34..66` → `--status-quarantine` amber + "Medium"; Low `<34` → `--status-pass` green + "Low"; Unscored → `--status-neutral` muted "—" (never a fabricated number).

> **Note on red:** the red hue serves three distinct semantics here — the Reject *action* (destructive button), the "Vacuous" *gate outcome* (`--status-fail`), and a "High" *risk tier* (`--status-fail`). All are the same actionable/attention semantic Phase 1 pre-reserved; each is always paired with a WORD ("Reject" / "Vacuous" / "High") and never conveyed by color alone (WCAG 1.4.1).

## Screen & State Inventory (Phase 6)

Supplementary contract for planner/executor. Every view ships every listed state (web e2e per 06-RESEARCH `scenarios.spec.ts`, against a mocked API, should touch them).

### Layout shell (all review views, inside the existing `<main className="p-6">`, single column, desktop-only dev tool)

1. **Page header block** (`flex items-center justify-between`, `lg` gap below): Left — Heading "Scenario review". (No global CTA here — generation is triggered from a flow, out of this UI's scope; the header carries only the title.)
2. **Filter bar** (list view only; `md` gap below header): a segmented control "Drafts · Approved · Rejected · All" built as plain accent-underlined segments over existing tokens (NOT a new tabs block — composition is sufficient; the active segment carries the accent indicator). The filter maps to the `?status=` query param so views are deep-linkable (`/scenarios?status=draft` default).
3. **Content region** (`xl` gap below the filter on list; `lg` on detail): the queue table, or the detail/review layout.

### 1. Review queue list (`/scenarios`)

- **Table** (secondary surface) columns: **Scenario** (feature/scenario title, Body; the accent drill-in link → `/scenarios/[id]`) · **Source flow** (flow name/category Body + mono flow-id caption) · **Risk** (the inherited risk badge: colored dot + mono score + tier word — surfaced so reviewers prioritize high-risk flows) · **Status** (the status badge: Draft amber / Approved green / Rejected muted; an "Edited" muted caption appears beside it when `edited=true`) · **Updated** (caption mono `updated_at` timestamp). Default sort: **risk descending, then most-recently-updated** (highest-risk drafts first — the actionable ordering).
- **States:** loading (skeleton rows) · **empty — no scenarios at all** (the "No scenarios yet" empty state centered in the table region) · **empty — filter has no matches** (the per-filter empty state, e.g. "No approved scenarios") · populated · rejected rows muted/de-emphasized · error (inline + Retry).

### 2. Scenario detail / review (`/scenarios/[id]`)

- **Breadcrumb caption** ("Scenario review / {title}") then the scenario title (Heading) + the **Status badge** + an "Edited" caption if applicable + the `created_at`/`updated_at` (mono captions).
- **Two-region body** (single column, stacked; `lg` gaps):
  - **"Gherkin" section** — the editable monospace `<textarea>` (styled-native, per Design System note) prefilled with `gherkin_text`. On the read state it shows the saved Gherkin; the reviewer edits in place. Below it: **"Save edits"** (enabled only when the text differs) + **"Cancel"** (reverts to saved). If the Gherkin fails the **syntax lint** on save, the parser error renders inline above the editor (mono detail) and the text is preserved.
  - **"Assertion checks" section** (secondary-surface card) — the per-`Then` no-vacuous results from `then_refs` (D-03): one row per `Then` showing the literal `Then` text (mono) + its gate indicator (green "Resolved" + the resolved `kg_ref` mono / red "Vacuous" + the reason mono / muted "Pending re-check" on an unsaved edit). A summary line above the list states all-pass or "{n} Then step(s) assert nothing recorded…". **Offending `Then`s are highlighted** (left red border + muted-red tint).
  - **"Source flow" section** — the flow name + category + its **risk badge** + an accent link "View in knowledge graph" → the flow's detail under `/graph/flows/[id]` (reusing the Phase-5 surface).
- **Action bar** (sticky or end-of-content; `md` gap): **"Approve scenario"** (accent primary — **disabled** unless the Gherkin parses AND every `Then` resolves AND there is no unsaved edit; a caption explains why when disabled) · **"Reject scenario"** (destructive variant → confirmation dialog) · (Save edits / Cancel live with the editor).
- **The edit-in-place flow (the defining interaction):**
  1. Reviewer edits the Gherkin → "Save edits" enables, Approve disables, affected `Then` indicators flip to muted "Pending re-check".
  2. "Save edits" → `POST /api/scenarios/{id}/edit` re-runs **both gates server-side** (syntax lint + no-vacuous). On **fail (422)** → the failure renders inline (parser error above the editor and/or the offending `Then`s flagged red in the gate panel), the row stays `draft`, `edited=true`, text preserved, **Approve stays disabled** — never a toast. On **success** → the gate panel repaints with the honest fresh results, `edited=true`, a "Edits saved" toast, and Approve enables iff all checks pass.
  3. "Approve scenario" → `POST /api/scenarios/{id}/approve` (re-runs both gates as defense-in-depth) → `status=approved`, "Scenario approved" toast, return to the queue. Only approved scenarios feed codegen (D-01).
- **States:** loading (skeleton editor + skeleton gate rows — never a premature green) · populated/read · editing (dirty) · save-failed (inline lint and/or vacuous failures, text preserved) · save-succeeded (fresh honest gate results) · approved (badge flips green, actions disabled/replaced with a muted "Approved" note) · rejected · not-found ("No scenario found." + breadcrumb link back to the queue) · error (inline + Retry) · graph-down during a re-check (inline "knowledge graph isn't reachable… `graph_mode up`" + Retry checks; Approve stays disabled).

### Shared addition

- **"Scenarios" sidebar nav item** — appended to `NAV_ITEMS` in `components/app-sidebar.tsx` following the documented flat-list `{icon, label, href}` contract; `icon: ListChecks`, `label: "Scenarios"`, `href: "/scenarios"`, active via `pathname.startsWith("/scenarios")`. Positioned after "Knowledge graph".

### Data fetching & boundary

- All views fetch via `@tanstack/react-query` against the review router (`GET /api/scenarios?status=`, `GET /api/scenarios/{id}`) through the existing same-origin `/api/:path*` rewrite, so the httpOnly auth cookie rides automatically (consistent with `lib/api/client.ts`). Mutations — `POST /api/scenarios/{id}/edit`, `/approve`, `/reject` — go through the existing `api` client wrapper (incl. `api.post`), with **list/detail query invalidation** on success (the Phase-1 mutation pattern; no optimistic updates — the gate result is authoritative and must come from the server). Responses are validated at the boundary with `zod` schemas in `lib/api/scenarios.ts` that mirror the Pydantic response models (06-RESEARCH `scenarios` + `then_refs` shapes).
- Endpoints are `Depends(get_current_user)`-gated server-side; an unauthenticated request returns 401 and the existing client wrapper redirects to `/login` (no per-page auth handling).
- **No fabricated gate state ever crosses the boundary:** the per-`Then` indicators render strictly from the server's `then_refs` resolution result; the client never computes or assumes a "Resolved" — green requires the server to have confirmed it (D-03 honesty).

### Accessibility

- **Tables:** real `<table>` semantics via shadcn `table` (`<th scope="col">`). The risk/status badges are NEVER color-only — each carries its WORD (tier / status) + (for risk) the numeric score as text alongside the colored dot (WCAG 1.4.1); the dot is `aria-hidden`.
- **Gate indicators:** the per-`Then` outcome is NEVER color-only — each carries the WORD ("Resolved" / "Vacuous" / "Pending re-check") and an icon with an accessible label, plus the mono `kg_ref`/reason as real text. Offending `Then`s carry both the border/tint AND the red "Vacuous" text.
- **Editable Gherkin area:** the `<textarea>` has an associated `<label>` ("Gherkin"), is keyboard-reachable with the 2px accent focus ring, and is a real resizable text control (not a contenteditable div). The save/approve disabled state uses the native `disabled` attribute with the explanatory caption as `aria-describedby`.
- **Approve gating:** the disabled "Approve scenario" exposes its reason via `aria-describedby` (the helper caption), so AT users learn why it's disabled, not just that it is.
- **Reject confirmation:** a shadcn `dialog` with focus trapped, the destructive button clearly labeled, and the dismiss ("Keep in queue") as the default focus.
- **Empty/error regions:** rendered as text with a reachable action link/button (Go to knowledge graph / View drafts / Retry / Retry checks); never an icon-only or color-only state.
- **Breadcrumbs:** an `aria-label="Breadcrumb"` nav with the current scenario marked `aria-current="page"`.
- **Toasts (sonner):** success-only and supplementary — every state they announce (saved/approved/rejected) is also reflected in the persistent UI (badge/queue), so a missed toast never hides state.
- **Motion:** no streaming/animation on this surface; nothing to gate on `prefers-reduced-motion` beyond inherited component defaults.

### Interaction defaults (inherited from 01-UI-SPEC, applied here)

- Focus rings: 2px accent ring on all interactive elements (links, filter segments, the Gherkin `<textarea>`, all buttons).
- Sentence case throughout; verbs-first on buttons ("Approve scenario", "Save edits", "Reject scenario", "Retry").
- Errors render inline in the content/gate region, never as toasts; success toasts (sonner, bottom-right) for the three mutations only.
