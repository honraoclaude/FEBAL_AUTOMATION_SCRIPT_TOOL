<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GHCR | The workflow authenticates and pushes images using the repo token |
| published image → K8s/runtime | The image artifact is consumed downstream (Plan 03) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-05 | Elevation of Privilege | GHCR token scope | mitigate | permissions: { contents: read, packages: write } only — the built-in GITHUB_TOKEN, never a PAT (RESEARCH Security V14). |
| T-11-06 | Information Disclosure | CI logs | mitigate | Never print the token / Authorization header in any step (the run-suite.yml T-07-07 precedent); docker/login-action handles auth without echoing. |
| T-11-07 | Tampering | image supply chain | mitigate | SHA-tagged immutable refs + reproducible `uv sync --frozen` / `npm ci` builds; the build-publish job runs only after the test gate passes (needs: test). |
| T-11-08 | Information Disclosure | published image contents | mitigate | Production images run prod servers (no dev hot-reload watchers); no secrets baked in — secrets are injected at runtime via env/K8s Secrets, never in the Dockerfile. |
| T-11-09 | Spoofing | publishing the wrong image | accept | saucedemo (test fixture) is explicitly excluded from the publish matrix (D-02); only api+web go to GHCR. |
</threat_model>
