# Phase 9 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase 9 UI surface (D-06 — MINIMAL):** ONE new authenticated section — the **Defects** review queue under `app/(dashboard)/defects/`. It is the human-in-the-loop surface that D-04's autonomy gate requires: a person reviews every machine-classified product defect and either **applies** it (files/updates a Jira issue) or **rejects** it, and reads the **calibration** numbers (QUAL-03 accuracy + draft precision) before flipping the autonomous-filing flag. It comprises two surfaces plus one shared panel:

1. **Defect review queue (list)** (`/defects`) — the section home and the JIRA-02 review-queue surface. A status-filter segment (Drafts · Applied · Rejected · All, default Drafts) + a class-filter segment (All · Infrastructure · Automation · Product defect) above a **drafts table**: each draft defect shows the **class badge** (Infrastructure / Automation / Product defect — visually AND lexically distinct), the **0-100 confidence** indicator, the source **test / flow / execution** refs, the defect **status** (draft/applied/rejected), and the optional linked **Jira key**. Rows drill into the detail view. Above the table sits the **calibration panel** (the shared component, below).
2. **Defect detail / review** (`/defects/[id]`) — the per-defect review surface. The full **proposed Jira issue** (summary, description prose, steps to reproduce, expected/actual, severity, priority) + the **cited evidence** (error type, DOM diff, healing history, infra health) + **attachment links** (screenshot / trace / video / logs) built as auth-gated URLs from run-relative basenames (NEVER raw filesystem paths) + the **fingerprint** + (if it exists) the linked **Jira key**. Action bar: **Apply** (file/update to Jira — shows an HONEST pending → result state, never fake-instant success) and **Reject** (destructive → confirm). When the fingerprint already matches an open Jira issue, Apply reads "Update {key}" (add a comment + re-attach evidence), NOT "Create".
3. **Calibration panel** (shared, rendered atop the queue) — the DEF-03 / QUAL-03 read-only display: measured **classification accuracy** vs the **>85% gate**, measured **draft precision** vs the **>90% gate**, the calibrated **confidence threshold**, and the **autonomous-filing flag** state (on/off). It surfaces — read-only — whether both gates are met and states clearly that **filing stays draft-only** until a human flips the flag via config. Honest empty state before any measurement exists.

**DEFERRED to Phase 10 (do NOT spec here):** the rich traceability-chain VISUALIZATION, the classification/defect-trend DASHBOARDS + analytics, RBAC, and Elasticsearch-backed defect search. This phase persists the classification + defect + link data and ships only the minimal review queue + calibration panel. There is no chart on this surface — calibration is a small number/gauge display, not a Recharts trend (a Phase-10 concern).

**Design system is LOCKED by 01-UI-SPEC.md, extended by 04/05/06/07-UI-SPEC, and confirmed in `apps/web/components.json` + the vendored `apps/web/components/ui/` on disk this session — it is REUSED verbatim here, not redefined.** This document specifies only what is NEW for the defects section: the page set, the queue table, the class/confidence/status treatments, the proposed-Jira-issue + evidence layout, the apply/update/reject flow with its honest pending state, the calibration panel, the artifact-link contract, every empty/loading/error/honest state, and the sidebar nav entry — all in the established voice. Every spacing/typography/color value below REFERENCES the Phase-1 tokens.

**AUTO-mode note:** No design-system questions were asked — 01-UI-SPEC (approved 2026-06-12; the locked tokens), 06-UI-SPEC (the Scenario Review Queue: the list+detail+apply/reject precedent this directly mirrors), 07-UI-SPEC (approved 2026-06-20; the artifact-link contract, the honest-pending-state discipline, the sidebar-append pattern, the styled-native-over-new-shadcn rule), 09-CONTEXT (D-01..D-06), and 09-RESEARCH (the defect/classification model, the draft-queue apply/reject lifecycle, the accuracy/precision calibration, the FakeJira contract, the evidence taxonomy) already answered every contract question. The codebase was scanned this session: `apps/web/components/ui/` (15 vendored blocks), `app/(dashboard)/scenarios/` + `executions/` (the list+detail patterns), `app/globals.css` (`--status-*` tokens), `components/app-sidebar.tsx` (the `NAV_ITEMS` contract), `routers/heals.py` (the auth-gated apply/reject API shape). Sources are cited per section; anything not sourced is a documented default.

No new palette, no new tokens.

| Role | Value | Usage on the defects section |
|------|-------|---------------------------------|
| Dominant (60%) | #09090B (zinc-950) | Page background / `<main>` surface |
| Secondary (30%) | #18181B (zinc-900), borders #27272A (zinc-800) | The calibration panel, the queue table surface + header row, the proposed-issue card, the evidence cards, the attachments panel, calibration tiles |
| Accent (10%) | #3B82F6 (blue-500) | See reserved list below |
| Destructive | #EF4444 (red-500) | The **Reject defect** control (destructive variant button) + its confirm dialog; shares the hue with `--status-fail` used for the Product-defect class accent and the rejected status |

Text: primary #FAFAFA (zinc-50); muted #A1A1AA (zinc-400) for table column headers, timestamps, ref/id captions, the fingerprint, rejected-row de-emphasis, and breadcrumbs.

**Accent reserved for:** the active "Defects" sidebar nav indicator; focus rings on all interactive elements (2px accent ring, inherited) including the filter segments; the **primary "Apply" CTA** on the detail view (the one accent action that files/updates the issue); the defect-row **drill-in link** from a queue row to its detail; the **Jira key link** (an `<a>` to the live issue once filed) and the "View in knowledge graph" / source-ref links; the "Back to defects" breadcrumb link. Never for borders, table rows at rest, class/status/confidence badges, the reject button, or calibration values.

**Defect class → status-token mapping (the visually + lexically distinct treatment — REUSE existing `--status-*` tokens, do NOT introduce new colors; class is ALWAYS conveyed by its WORD + a distinct icon, never color alone):**

| Class | Token | Resolved | Indicator treatment | Icon (lucide) |
|-------|-------|----------|---------------------|---------------|
| **Infrastructure** | `--status-neutral` (#A1A1AA zinc-400) | muted | muted badge + "Infrastructure" (Label) — environmental noise, not a product fault → de-emphasized, never alarming | `ServerCog` |
| **Automation** | `--status-quarantine` (#F59E0B amber-500) | amber | amber badge + "Automation" (Label) — a test/locator problem to fix in the suite, the same "needs-attention but not a product bug" semantic as a flaky/quarantine signal | `Wrench` |
| **Product defect** | `--status-fail` (#EF4444 red-500) | red | red badge + "Product defect" (Label) — a real bug in the app under test, the actionable/attention semantic; the only class eligible for autonomous Jira filing | `Bug` |

> **The single most important color decision of this phase:** the three classes are distinct in hue AND word AND icon. **Product defect is red** (a real bug — the actionable signal), **Automation is amber** (fix the suite — needs attention, not a product fault), **Infrastructure is muted/neutral** (environmental noise). This mirrors the Phase-7 "Failed=red / Flaky=amber / Aborted=muted" semantic exactly — a class is NEVER conveyed by color alone (WCAG 1.4.1); the word and icon always accompany the hue.

**Confidence (0-100) → band color (the meter fill; the band edges read from the calibrated `jira_confidence_threshold`, NEVER a hardcoded literal — the heal-band/threshold precedent):**

| Band | Token | Treatment |
|------|-------|-----------|
| At/above the calibrated threshold (autonomous-filing-eligible if the class is Product defect) | `--status-pass` (#22C55E green-500) | green meter fill + the mono numeral; this defect's confidence clears the gate |
| Below the threshold | `--status-quarantine` (#F59E0B amber-500) | amber meter fill + the mono numeral; classified but below the autonomous-filing floor (stays human-reviewed) |

> The confidence band is computed against `settings.jira_confidence_threshold` served on the API payload — the UI renders the band the server reports, never a client-invented cutoff. Below-threshold is amber (caution: machine is less sure), not red (red is reserved for the Product-defect CLASS, a different axis). The numeral is the source of truth; the color is supplementary.

**Defect status → status-token mapping (the lifecycle; ALWAYS paired with its word):**

| Status | Token | Resolved | Treatment |
|--------|-------|----------|-----------|
| Draft | `--status-quarantine` (#F59E0B amber-500) | amber | amber badge + "Draft" — awaiting human apply/reject (the review-queue default; mirrors the Scenario "Draft" + the heal "quarantine" awaiting-review semantic) |
| Applied | `--status-pass` (#22C55E green-500) | green | green badge + "Applied" + the mono Jira key link (filed/updated to Jira) |
| Rejected | `--status-neutral` (#A1A1AA zinc-400) | muted | muted badge + "Rejected"; the row reads muted (a deliberate human decision — not an error → muted, not red) |

**Calibration gate → met/unmet display (read-only; the >85% accuracy + >90% precision gates):**

| State | Token | Treatment |
|-------|-------|-----------|
| Gate met (measured ≥ the threshold) | `--status-pass` (#22C55E green-500) | green check + "Met" beside the measured value vs its target |
| Gate not yet met (measured < the threshold) | `--status-quarantine` (#F59E0B amber-500) | amber dot + "Not met yet" — filing stays draft-only (caution, not failure → amber, never red) |
| Not yet measured (no labeled-set run) | `--status-neutral` (#A1A1AA zinc-400) | muted "—" + the honest "Run the accuracy harness to measure." copy |

> **Note on red:** red serves two distinct semantics here — the Reject *action* (destructive button) and the Product-defect *class* (`--status-fail`). Both are the actionable/attention semantic Phase 1 pre-reserved; each is always paired with a WORD ("Reject defect" / "Product defect") and never color alone (WCAG 1.4.1). **A below-threshold confidence and an unmet gate are AMBER, never red** — they are caution states (the machine is less sure / the numbers aren't in yet), not failures.

## Screen & State Inventory (Phase 9)

Supplementary contract for planner/executor. Every view ships every listed state (web e2e against a mocked API — mirroring the `scenarios` review-queue e2e; no real Jira, no provider keys, all states mockable).

### 1. Defect review queue (list) (`/defects`)

The Apply/Reject mutations (`POST /api/defects/{id}/apply`, `POST /api/defects/{id}/reject`) go through the existing `api` client wrapper with list + detail + calibration query invalidation on success (NO optimistic updates — defect status, the Jira key, and the dedup result are server-authoritative).

### Accessibility

- **Tables:** real `<table>` semantics via shadcn `table` (`<th scope="col">`). Class, status, and calibration-gate badges are NEVER color-only — each carries its WORD (class name / status / "Met"/"Not met yet") alongside the colored dot/icon (WCAG 1.4.1); the dot/icon is `aria-hidden`. **The three classes are conveyed by the WORDS "Infrastructure"/"Automation"/"Product defect" AND distinct icons**, not by muted-vs-amber-vs-red alone.
- **Confidence meter:** a real `role="progressbar"` with `aria-valuenow`/`aria-valuemin="0"`/`aria-valuemax="100"` and an accessible name combining the value and whether it clears the threshold (e.g. `aria-label="Confidence 82 of 100, above the filing threshold"`) — never a decorative-only bar; the mono numeral is always present as text.
- **Calibration tiles:** each tile's value has an `aria-label` combining label + value + gate state (e.g. `aria-label="Classification accuracy 88 percent, target 85 percent, met"`). The autonomy-flag indicator is `role="status"` text ("Autonomous filing: Off"), not color-only.
- **Apply control:** a real `<button>`; while filing it is natively `disabled` with its "Filing…" label as the accessible state (`aria-busy` while pending). The "create" vs "update {key}" label difference is in the accessible name so a screen-reader user knows whether a new issue is created or an existing one updated.
- **Reject control:** a real `<button>`; the confirmation is a shadcn `dialog` with focus trapped and "Keep in queue" as the default focus.
- **Attachment links:** real `<a>`/`<Link>` with descriptive accessible names ("Trace for defect {id}"); the absent-artifact state is real text ("Video captured on failure only."), not an empty or dead link.
- **Jira key link:** a real `<a>` with an accessible name ("Open Jira issue {key}"); opens the live issue once filed.
- **Empty/error regions:** rendered as text with a reachable action link/button (Go to executions / View drafts / Retry); never icon-only or color-only.
- **Motion:** any meter/badge entrance animation respects `prefers-reduced-motion` (drop to instant when set), mirroring the inherited motion rule.
