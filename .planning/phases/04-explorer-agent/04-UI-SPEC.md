# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase 4 UI surface:** exactly ONE new authenticated page — the **Live Exploration View** at `app/(dashboard)/explore/[runId]/page.tsx` (EXPL-01). It is reached after starting an exploration of a registered target. The page subscribes to a same-origin SSE stream (`GET /api/explore/{runId}/events`, D-07) and renders live progress: a header **counters** strip, a scrolling per-step **action feed**, and a **current-page panel** (title/URL + latest screenshot thumbnail) (D-08). Plus two small shared additions the page requires: an "Explore" launch affordance on the existing targets row-actions menu, and an "Explorations" sidebar nav item.

**Design system is LOCKED by 01-UI-SPEC.md and confirmed in `apps/web/components.json` — it is REUSED verbatim here, not redefined.** This document specifies only what is NEW for the live view: its layout, its states, streaming/growth behavior, reconnection, accessibility, and the copy in the established voice. Every spacing/typography/color value below REFERENCES the Phase-1 tokens.

**AUTO-mode note:** No design-system questions were asked — 01-UI-SPEC (approved 2026-06-12), 04-CONTEXT D-07/D-08, and 04-RESEARCH (the `ExploreProgressEvent` schema, SSE seam, screenshot paths) already answered every contract question. Sources are cited per section; anything not sourced is a documented default.

## Color

Inherited verbatim from 01-UI-SPEC dark-first zinc palette. No new palette. The four `--status-*` tokens already defined in `app/globals.css` carry the exploration run states — that is exactly why they were defined ahead of need in Phase 1.

| Role | Value | Usage on the live view |
|------|-------|------------------------|
| Dominant (60%) | #09090B (zinc-950) | Page background / `<main>` surface |
| Secondary (30%) | #18181B (zinc-900), borders #27272A (zinc-800) | Counter tiles, the action-feed panel card, the current-page panel card, the screenshot thumbnail frame |
| Accent (10%) | #3B82F6 (blue-500) | See reserved list below |
| Destructive | #EF4444 (red-500) | "Stop exploration" confirm button + the failed/budget-exhausted terminal state |

Text: primary #FAFAFA (zinc-50); muted #A1A1AA (zinc-400) for counter labels, step indices, timestamps, URL captions, and refused/skipped feed lines.

**Accent reserved for:** the primary "Start exploration" button (on the targets page / launch affordance), focus rings on all interactive elements (2px accent ring, inherited interaction default), the active "Explorations" sidebar nav indicator, and inline links (e.g. "Back to targets", "View knowledge graph" once Phase 5 lands). Never for borders, counter values, feed text, badges, the screenshot frame, or the connection/status pill.

**Run-state status colors (REUSE the existing `--status-*` tokens — do NOT introduce new colors):**

| Live-view state | Token | Resolved | Where it renders |
|-----------------|-------|----------|------------------|
| Connecting / running | `--status-pass` (#22C55E green-500) used as a steady dot for "live" + a subtle pulse animation on the status pill | green | Status pill dot while the stream is open and the run is `running` |
| Converged / done (success) | `--status-pass` (#22C55E) | green | Terminal banner "Exploration complete" + final status pill |
| Failed | `--status-fail` (#EF4444, shares destructive) | red | Terminal banner "Exploration failed" + status pill |
| Budget exhausted | `--status-quarantine` (#F59E0B amber-500) | amber | Terminal banner "Stopped — budget reached" (a bounded, expected stop, NOT a failure → amber, not red) |
| Stopped by user (kill switch) | `--status-neutral` (#A1A1AA zinc-400) | muted | Terminal banner "Exploration stopped" |
| Reconnecting (stream dropped, EventSource retrying) | `--status-quarantine` (#F59E0B amber) | amber | Status pill switches to amber "Reconnecting…" while EventSource is in its retry backoff |
| Refused action (risk gate / off-origin) in the feed | `--status-quarantine` (#F59E0B amber) icon + muted text | amber/muted | A `ShieldOff`/`Ban` lucide icon on the refused feed row; the row text is muted with an amber leading icon — distinct from a normal action row |

Rationale for reuse: green/amber/red/neutral were pre-reserved in Phase 1 as pass/quarantine/fail/neutral semantics precisely so agent run states inherit them with zero new color decisions. "Budget reached" is amber not red because it is a designed, successful termination (D-05 saturation/cap), not an error.

## Screen & State Inventory (Phase 4)

Supplementary contract for planner/executor. The page ships every listed state.

### Live Exploration View (`app/(dashboard)/explore/[runId]/page.tsx`)

**Layout (desktop-only developer tool, single column inside the existing `<main className="p-6">`):**

1. **Page header block** (`flex items-center justify-between`, `lg` gap below):
   - Left: Heading "Exploring {target name}" + caption "Run {run_id}" (mono) beneath it; the **status pill** sits inline to the right of the title.
   - Right: "Stop exploration" destructive button (running only); a muted "Back to targets" inline link.
2. **Counters strip** (`xl` gap below header): a row of 4 counter tiles (`md` gap between, each a secondary-surface card with `md` padding). Each tile: caption label (12px muted) on top, Label-weight value (14px/600) below; cost and elapsed values are mono.
   - Pages found — integer (`pages_found`)
   - Actions — integer (`actions_taken`)
   - Cost — `$0.0123` mono (`cost_usd`; sourced from the gateway's per-run counter, D-06 — the live view never computes spend)
   - Elapsed — `02:14 / 10:00` mono (`elapsed_s` vs the run's wall-clock budget)
3. **Panels grid** (`lg` gap): two panels side by side on wide viewports, stacked on narrow.
   - **Action feed panel** (flex-grow, secondary card): heading "Action feed"; a scrolling list of feed rows. Each row (`sm` vertical padding, `xs` icon gap): a small lucide icon for the verb (`Navigation`, `MousePointerClick`, `FileText`, `Type`, `Ban`/`ShieldOff` for refused), a caption step index + timestamp (mono, muted), and the Body action line (verb in sans, the target/element/URL in mono). Newest row appended at the bottom; the list auto-scrolls to the latest row UNLESS the user has scrolled up (see streaming behavior).
   - **Current-page panel** (fixed/secondary card): heading "Current page"; the current page **title** (Body) + **URL** (caption, mono, truncated with title attribute) + the latest **screenshot thumbnail** in a 16:10 framed box (secondary border). The thumbnail `src` is the run-scoped screenshot path (`workspaces/{run_id}/...` served via an authenticated `/api/explore/{run_id}/screenshot/...` route, or an API-provided URL on the event); it swaps to the newest `screenshot_path` as events arrive, with a short cross-fade (CSS opacity, ≤150ms) to avoid a hard flash.

**States (every state must be implemented; Playwright e2e should touch them via a mocked SSE stream):**

- **connecting** — page mounted, `EventSource` opening, no events yet. Status pill "Connecting…" (muted/green dot, no pulse). Counters render zeros via skeleton-less placeholders ("—" for cost/elapsed). Feed shows "Waiting for the first step…". Screenshot frame shows "No screenshot yet".
- **running (live)** — stream open, events flowing. Status pill "Live" green with a subtle pulse. Counters update on each event. Feed appends rows + auto-scrolls. Screenshot swaps. "Stop exploration" button visible.
- **reconnecting** — `EventSource` fired `onerror` and is in its native retry backoff (readyState CONNECTING). Status pill "Reconnecting…" amber. The feed/counters freeze at last-known values (NOT cleared). No data is invented. On reconnect, the server re-sends a current-state snapshot event first (RESEARCH SSE section) which reconciles counters/feed-tail without a full replay.
- **converged / done** — terminal `stop_reason` in (`saturation`/`converged`) → green "Complete" pill + complete banner above the panels; "Stop exploration" button removed; `EventSource` closed by the client on terminal.
- **failed** — terminal `failed` → red "Failed" pill + failed banner with the error and the `docker compose ps` next-step copy.
- **budget-exhausted** — terminal `stop_reason == "budget"` → amber "Budget reached" pill + budget banner with the raise-overrides next step.
- **stopped (kill-switched)** — user confirmed "Stop exploration"; terminal `stopped` → neutral "Stopped" pill + stopped banner.
- **stream-lost (unrecoverable)** — `EventSource` exhausted retries AND the run is not known-terminal → inline "Lost connection…" message + "Reload" button. (Distinct from the amber transient "reconnecting".)
- **unknown run / 404** — the `runId` route param does not resolve (no such run) → centered message "No exploration found for this run." + "Back to targets" link. (Detected via the existing `GET /api/executions/{run_id}` poll returning 404 before/instead of opening the stream.)

### Shared additions (small, required by this page)

1. **Targets row-actions "Explore" item** — added to the existing `DropdownMenu` in `components/targets/targets-table.tsx`, placed above "Edit". Selecting it starts a run (`POST /api/explore` with `target_id`), fires the "Exploration started" toast, and navigates to `/explore/{run_id}`. Disabled (muted, with tooltip "Activate this target to explore it") for inactive targets. Uses the already-installed `tooltip` component.
2. **"Explorations" sidebar nav item** — appended to `NAV_ITEMS` in `components/app-sidebar.tsx` following the documented flat-list `{icon, label, href}` contract; `href: "/explore"` with active state derived from `pathname.startsWith`. (A run-less `/explore` index/listing page is OUT of scope for this phase — the nav item may point at the most-recent run or render a thin "Start an exploration from the Targets page" placeholder; planner's discretion, kept minimal.)

### Streaming, growth-capping & reconnection behavior

- **Transport:** browser `EventSource('/api/explore/{runId}/events')` — same-origin via the existing Next `/api/:path*` rewrite (`next.config.ts`), so the httpOnly auth cookie rides automatically (no token handling in the page, consistent with `lib/api/client.ts`). EventSource cannot set headers; cookie auth is the only auth — the SSE route is `Depends(get_current_user)` server-side.
- **Event parsing:** each SSE `message`/`step` event `data` is JSON parsed into the `ExploreProgressEvent` shape (run_id, step, pages_found, actions_taken, current_url, current_title, screenshot_path, feed_line, cost_usd, elapsed_s, stop_reason). Counters take the latest event's absolute values (not deltas); the feed appends `feed_line`.
- **Feed growth cap:** the action feed is capped to the **last 200 rows** in the DOM (a long run can emit hundreds of steps; unbounded DOM growth is the failure mode). Older rows are dropped from the rendered list (kept only in the latest-200 window). A caption above the feed notes "Showing the latest 200 steps" once the cap is hit. (Virtualization via a windowing lib is explicitly NOT added — a 200-row cap with plain rendering is simpler and sufficient for a single-run dev tool; documented default.)
- **Auto-scroll discipline:** the feed auto-scrolls to the newest row only when the user is already at the bottom. If the user scrolls up to read history, auto-scroll pauses and a small "Jump to latest" affordance appears (accent inline link/button); clicking it resumes auto-scroll. Prevents yanking the viewport while reading.
- **Screenshot update:** on each event carrying a new `screenshot_path`, swap the `<img src>` with a ≤150ms opacity cross-fade; keep the previous image visible until the new one loads (`onLoad`) to avoid a blank flash. Always show the LATEST captured state only (no carousel this phase).
- **Reconnection:** rely on EventSource's native auto-reconnect (it reissues with `Last-Event-Id`). The page does NOT tear down state on `onerror`; it shows the amber "Reconnecting…" pill and waits. The server re-emits a current-state snapshot on (re)subscribe so counters/screenshot/feed-tail reconcile without replaying the whole run. If the run is already terminal when the stream (re)opens, the snapshot carries the terminal `stop_reason` and the page renders the terminal state directly.
- **Terminal close:** on any terminal `stop_reason`, the page calls `eventSource.close()` and renders the matching banner + pill; it does not keep a dangling connection.

### Accessibility

- **Live region:** the action feed container is an ARIA live region — `role="log"` `aria-live="polite"` `aria-relevant="additions"` — so each appended action line is announced to screen readers without stealing focus. (Polite, not assertive — a fast stream must not flood/interrupt.)
- **Status pill:** carries `role="status"` `aria-live="polite"` so connection/run-state changes ("Live" → "Reconnecting…" → "Complete") are announced.
- **Counters:** each tile's value has an `aria-label` combining label + value (e.g. `aria-label="Pages found: 12"`) since the visual label/value split is two elements.
- **Refused rows:** the amber leading icon is `aria-hidden`; the row text itself carries the full meaning ("Refused … — destructive action blocked"), so the refusal reason is not color-only (WCAG 1.4.1 — never rely on color alone; the amber is reinforced by the icon AND the word "Refused").
- **Screenshot `alt`:** the thumbnail `alt` is descriptive and dynamic — `alt="Latest captured screenshot: {current_title}"`; the "No screenshot yet" empty state is real text, not an empty `alt`.
- **Stop button:** the destructive "Stop exploration" control is a real `<button>` with the confirm dialog trapping focus (reuse the `DeactivateDialog`/Dialog pattern — focus management already correct in the installed dialog).
- **Keyboard:** all interactive elements (Stop, Back to targets, Jump to latest, terminal links, Reload) reachable by Tab with the 2px accent focus ring (inherited interaction default). No interaction is hover-only.
- **Motion:** the "Live" pulse and the screenshot cross-fade respect `prefers-reduced-motion` (drop the pulse/fade to instant when set).

### Interaction defaults (inherited from 01-UI-SPEC, applied here)

- Focus rings: 2px accent ring on all interactive elements.
- Toasts (sonner, bottom-right): success only — "Exploration started". All stream/terminal errors render inline (banner), never as toasts.
- The page does NOT poll for status in parallel with the stream during a live run (the stream is the source of truth); it MAY use `GET /api/executions/{run_id}` once on mount to resolve a 404 (unknown run) and to render the terminal state immediately if the user opens the page after the run already finished.
