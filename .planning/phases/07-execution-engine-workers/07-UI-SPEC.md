# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase 7 UI surface:** the **Executions** section — a small set of new authenticated pages under `app/(dashboard)/executions/` that turn the Phase-6-owned Playwright automation into a watchable, history-backed run-at-scale engine. It comprises three surfaces:

1. **Run launcher + history list** (`/executions`) — the section's home. A compact **run launcher** (pick a tier → "Start run") sits above the **runs history table** (tier, started, duration, pass/fail/flaky counts, status) plus the **trend charts** (pass-rate over time, durations) via Recharts. This is the EXEC-05 surface.
2. **Live execution view** (`/executions/[runId]`) — the EXEC-06 surface, reached after starting a run (or by opening a still-running run from the history table). Per-**test** progress rows (test id/name, status, attempt count, duration), an overall **progress + counts** strip taking absolute values from the latest SSE event, and a prominent **graceful Kill** control. Mirrors the Phase-4 Live Exploration View verbatim, retargeted from per-step exploration events to per-test execution events.
3. **Run detail (terminal)** (`/executions/[runId]` once terminal) — the SAME route renders a terminal run-detail layout: the final per-test results table with **flaky vs failed** visually distinct, attempt counts, durations, and **auth-gated artifact links** (screenshot / trace / video) built from run-relative basenames — never raw filesystem paths.

The defining interactions are: **launch a tier run** (EXEC-01), **watch it live with honest server-authoritative per-test status + a graceful kill that shows a real "stopping…" draining state** (EXEC-06 / D-07), and **review history + flaky detection** (EXEC-05) where a test that passed-on-retry is flagged as flaky (infra flake) distinctly from a product failure that failed all attempts (D-05).

**Design system is LOCKED by 01-UI-SPEC.md, extended by 04/05/06-UI-SPEC, and confirmed in `apps/web/components.json` + the vendored `apps/web/components/ui/` on disk — it is REUSED verbatim here, not redefined.** This document specifies only what is NEW for the executions section: the page set, the launcher, the runs table + trend charts, the live per-test view, the graceful-kill draining state, the artifact-link contract, the flaky-vs-failed treatment, every empty/loading/error/honest state, and the sidebar nav entry — all in the established voice. Every spacing/typography/color value below REFERENCES the Phase-1 tokens.

**AUTO-mode note:** No design-system questions were asked — 01-UI-SPEC (approved 2026-06-12), 04-UI-SPEC (the Live Exploration View this directly mirrors), 06-UI-SPEC (the most recent approved precedent), 07-CONTEXT (D-01..D-08), and 07-RESEARCH (the per-test event shape, the `test_runs`/`test_results`/`test_artifacts` model, the trend/duration/flaky queries, the graceful kill drain, the artifact layout) already answered every contract question. Sources are cited per section; anything not sourced is a documented default.

## Color

Inherited verbatim from 01-UI-SPEC dark-first zinc palette, reusing the **same `--status-*` tokens** Phase 1 pre-reserved and Phases 4/5/6 bound — now applied to (a) the **run status** lifecycle, (b) the **per-test verdict** (passed / flaky / product-failure / aborted), and (c) the **kill/draining** transient state. No new palette, no new tokens.

| Role | Value | Usage on the executions section |
|------|-------|---------------------------------|
| Dominant (60%) | #09090B (zinc-950) | Page background / `<main>` surface; chart plot background |
| Secondary (30%) | #18181B (zinc-900), borders #27272A (zinc-800) | Counter tiles, the launcher card, the runs table surface + header row, the per-test list panel, the trend-chart cards, the artifact-links panel |
| Accent (10%) | #3B82F6 (blue-500) | See reserved list below |
| Destructive | #EF4444 (red-500) | The **Kill run** control (destructive variant button) + its confirmation; shares the hue with `--status-fail` used for the "Failed" verdict and the failed run status |

Text: primary #FAFAFA (zinc-50); muted #A1A1AA (zinc-400) for table column headers, timestamps, counter labels, run-id sub-labels, aborted/skipped de-emphasis, chart axis ticks, and breadcrumbs.

**Accent reserved for:** the active "Executions" sidebar nav indicator; focus rings on all interactive elements (2px accent ring, inherited) including the tier picker; the **primary "Start run" CTA** on the launcher (the one accent action on the list view); the run-id / "View run" **drill-in link** from a history row to the live/detail view; the "Back to executions" breadcrumb link; the "Jump to latest" affordance on the live test list; and the **primary trend-line series** color in the pass-rate chart (one accent series; the rest of the chart uses `--status-*`/muted). Never for borders, table rows at rest, icons-at-rest, status/verdict badges, the kill button, or counter values.

**Run status → status-token mapping (REUSE the existing `--status-*` tokens — do NOT introduce new colors):**

| Run status | Token | Resolved | Treatment |
|-----------|-------|----------|-----------|
| Queued | `--status-neutral` (#A1A1AA zinc-400) | muted | badge muted dot + "Queued" (enqueued, no test started yet) |
| Running | `--status-pass` (#22C55E green-500) as a steady "live" dot + subtle pulse on the live pill | green | live pill "Running" while the SSE stream is open and tests are in flight |
| Stopping (draining — D-07) | `--status-quarantine` (#F59E0B amber-500) | amber | live pill "Stopping…" while the kill flag is set and the worker drains the in-flight test (an HONEST transient — never a fake-instant kill) |
| Passed | `--status-pass` (#22C55E green-500) | green | terminal banner "Run complete — all tests passed" + green run badge |
| Failed | `--status-fail` (#EF4444 red-500) | red | terminal banner "Run complete — {n} failed" + red run badge (at least one product failure) |
| Passed with flakes | `--status-pass` green badge + a `--status-quarantine` amber "flaky" sub-chip | green + amber | a run with zero product failures but ≥1 flaky test reads as Passed (green) with an amber "{n} flaky" chip — flakes are infra noise, not a run failure |
| Killed (user kill switch) | `--status-neutral` (#A1A1AA zinc-400) | muted | terminal banner "Run stopped" + muted run badge (a deliberate, expected stop — not an error → muted, not red) |

**Per-test verdict → status-token mapping (the HONEST, server-authoritative flaky display — D-05):**

| Verdict | Token | Resolved | Indicator treatment |
|---------|-------|----------|---------------------|
| Passed (first attempt) | `--status-pass` (#22C55E green-500) | green | green `CheckCircle2` icon + "Passed" (Label). No attempt chip (or "1 attempt"). |
| Flaky (passed on a retry — infra flake) | `--status-quarantine` (#F59E0B amber-500) | amber | amber `RefreshCw`/`AlertTriangle` icon + "Flaky" (Label) + a mono "{n} attempts" chip. **Visually distinct from both Passed and Failed** — amber, never green, never red — because it passed but needed a retry (D-05). |
| Failed (product failure — all attempts failed) | `--status-fail` (#EF4444 red-500) | red | red `XCircle` icon + "Failed" (Label) + a mono "{n} attempts" chip. The row is highlighted (left red border + muted-red tint), mirroring the Phase-6 offending-`Then` treatment. |
| Running / queued (live) | `--status-pass` green (running, with pulse) / `--status-neutral` muted (queued) | green / muted | live in-flight rows show a green spinner/pulse "Running" or a muted "Queued"; absolute status comes from the latest SSE event only — never optimistically completed. |
| Aborted (drained by the kill switch) | `--status-neutral` (#A1A1AA zinc-400) | muted | muted `MinusCircle` icon + "Aborted" — a test that was queued/skipped when the run was killed. Distinct from Failed (it never ran to a product verdict). |

> **Note on red:** the red hue serves three distinct semantics here — the Kill *action* (destructive button), the "Failed" *verdict* (`--status-fail`), and the failed *run status* (`--status-fail`). All are the same actionable/attention semantic Phase 1 pre-reserved; each is always paired with a WORD ("Kill run" / "Failed") and never conveyed by color alone (WCAG 1.4.1). **Flaky is amber, never red** — the single most important color decision of this phase: a flake is infra noise that passed, NOT a product failure.

## Screen & State Inventory (Phase 7)

Supplementary contract for planner/executor. Every view ships every listed state (web e2e against a mocked API + a mocked SSE stream should touch them, mirroring `explore`/`scenarios` e2e).

### Layout shell (all executions views, inside the existing `<main className="p-6">`, single column, desktop-only dev tool)

1. **Page header block** (`flex items-center justify-between`, `lg` gap below): Left — Heading ("Executions" on list; "Running {tier} suite" / "{tier} run" on the run view) + the mono "Run {run_id}" caption + (live view only) the **live pill** inline to the right of the title. Right — (live view only) the destructive "Kill run" button while queued/running; a muted "Back to executions" breadcrumb link.
2. **Content region** (`xl` gap on list; `lg` on the run view).

### 1. Run launcher + history list (`/executions`)

- **Launcher card** (`md` gap below header, secondary surface, `md` padding): a "Run a suite" heading + the **tier picker** (styled-native `<select>` or segmented control — "Smoke / Sanity / Regression / Full / Risk-based", default Smoke) + the **resolved-count helper** caption (the honest count, or "selected when the run starts" for risk-based) + the accent **"Start run"** CTA. On start → `POST /api/executions` `{tier}` → "Run started" toast → navigate to `/executions/{run_id}`.
- **History table** (secondary surface) columns: **Tier** (Body) · **Started** (mono caption `started_at`) · **Duration** (mono `00:42`, or "—" while running) · **Results** (mono "{passed} passed · {failed} failed · {flaky} flaky" — flaky amber, failed red, each with its word) · **Status** (the run-status badge; a still-running run shows a green "Running" badge and its row drills into the LIVE view). Default sort: most-recently-started first.
- **Trends region** (`lg` gap below the table): two Recharts cards — **"Pass rate over time"** (a line/area series of `sum(passed)/sum(total)` per day from `test_runs`, accent primary series) and **"Run duration over time"** (a line/bar of run durations). Axis numerals mono; series colored with the accent (pass-rate) and a muted/`--status` hue (duration). A "flaky leaderboard" (top flaky `flow_id`s) MAY render as a small companion table beneath the charts (planner's discretion; data is the EXEC-05 flaky query).
- **States:** loading (skeleton table rows + skeleton chart blocks) · **empty — no runs** ("No runs yet" message; launcher stays usable; charts show "Trends appear after your first run.") · populated · running-row (green "Running" badge, drills to live view) · error (inline + Retry) · queue-down on start (inline launcher error with the `--profile queue up` next step).

### 2. Live execution view (`/executions/[runId]` while non-terminal)

**Layout (single column inside `<main className="p-6">`, mirrors the Phase-4 Live Exploration View):**

1. **Page header block** — Heading "Running {tier} suite" + caption "Run {run_id}" (mono) + the **live pill** inline; right: "Kill run" destructive button (queued/running only) + "Back to executions" link.
2. **Counters strip** (`xl` gap below header): a row of counter tiles (`md` gap, each secondary card, `md` padding) — **Passed** · **Failed** · **Flaky** · **Total** · **Elapsed** (mono `00:42`). Each tile: caption label on top, Label-weight value below; counts are absolute values from the latest SSE event (NOT deltas), mirroring the Phase-4 absolute-counter contract. A thin **progress affordance** (caption "{completed} of {total} tests" + a token-styled native progress bar — `bg-secondary` track, `--status-pass` fill; the shadcn `progress` block is NOT vendored, so this is a styled-native composition over tokens, consistent with the styled-native discipline) sits beneath the tiles.
3. **Per-test list panel** ("Test results", secondary card, `lg` gap below counters): a list of per-test rows. Each row (`sm` vertical padding, `xs` icon gap): the verdict icon, the test/scenario **name** (Body, sans), the **test id** (mono caption beneath), the **verdict word** (Passed/Flaky/Failed/Running/Queued/Aborted), a mono **"{n} attempts"** chip when attempts > 1, and the test **duration** (mono `1,240 ms`) once resolved. Newest-updated row behavior follows the Phase-4 feed: the list is bounded to a sensible cap (e.g. last 200 in-DOM if a suite is huge; a full suite is normally far smaller) and **auto-scrolls to the active test only when the user is at the bottom**, with a "Jump to latest" accent affordance when scrolled up.

**States (every state implemented; e2e via a mocked SSE stream):**

- **connecting** — page mounted, `EventSource` opening, no events yet. Pill "Queued"/"Connecting" (muted). Counters render zeros (`—` for elapsed). Test list shows "Waiting for the first test…".
- **running** — stream open, per-test events flowing. Pill "Running" green with a subtle pulse. Counters + progress update on each event (absolute). Test rows flip queued → running → passed/flaky/failed. "Kill run" visible.
- **stopping (draining)** — after the user confirms Kill: pill "Stopping…" amber, the inline draining banner ("Stopping the run… finishing the current test and dropping the rest."), the Kill button disabled showing "Stopping…". In-flight test drains to a verdict; remaining queued tests resolve to "Aborted". **No fake-instant kill** (D-07).
- **reconnecting** — `EventSource` `onerror` + native retry backoff. Pill "Reconnecting…" amber; counters/test list freeze at last-known values (NOT cleared, nothing invented). On reconnect the server re-sends a current-state snapshot that reconciles counters + the test-list tail without a full replay (mirrors the Phase-4 reconnection contract).
- **terminal: passed** — pill "Complete" green + the passed terminal banner (with the flaky sub-note if any); Kill removed; `EventSource` closed; the view becomes the run-detail layout (§3).
- **terminal: failed** — pill "Complete" + the failed terminal banner; failed test rows highlighted; the view becomes run-detail with artifacts.
- **terminal: killed** — pill "Stopped" muted + the killed banner; aborted rows muted; run-detail with the partial results kept.
- **stream-lost (unrecoverable)** — EventSource exhausted retries AND run not known-terminal → inline "Lost connection…" + "Reload".
- **unknown run / 404** — `runId` doesn't resolve → "No run found for this id." + "Back to executions" (detected via a single `GET /api/executions/{run_id}` on mount, as in Phase 4).

### 3. Run detail (terminal) (`/executions/[runId]` once terminal)

The SAME route, once the run is terminal, renders the run-detail layout (the live view "freezes" into it — no separate route):

- **Header**: "{tier} run" + "Run {run_id}" + the terminal run-status badge + mono `started_at` / `finished_at` / duration captions.
- **Summary counters** (the same tiles, now final/static): Passed · Failed · Flaky · Total.
- **"Test results" table** (secondary surface) columns: **Test** (name + mono id) · **Verdict** (the per-test verdict indicator — flaky amber, failed red, each with its word, failed rows highlighted) · **Attempts** (mono) · **Duration** (mono `1,240 ms`) · **Artifacts** (the artifact links for that test). Default sort: failures first, then flaky, then passed (the actionable ordering).
- **Artifacts** per test: **Screenshot / Trace / Video / Console log / Network log** links. Each is an **auth-gated URL built from the run-relative basename** — the client constructs `/api/executions/{run_id}/artifacts/{flow_id}/{kind}` (or an API-provided URL on the result payload) reusing the Phase-4 M-1 screenshot-route pattern (`screenshotUrl` analogue + the router-level containment guard that rejects `..`/separators and resolves inside the run dir). **Never a raw filesystem path.** Video is absent for passed/flaky tests → the honest "Video captured on failure only." caption (D-04), not a broken link.
- **States:** loading (skeleton table + skeleton artifact slots) · populated · a test with no failure (no video slot, honest caption) · error loading detail (inline + Retry) · 404.

### Data fetching, streaming & boundary

- **List + detail** fetch via `@tanstack/react-query` against the execution router (`GET /api/executions` for the history + trends, `GET /api/executions/{run_id}` for run detail) through the existing same-origin `/api/:path*` rewrite, so the httpOnly auth cookie rides automatically (consistent with `lib/api/client.ts`). The launcher mutation `POST /api/executions {tier}` and `POST /api/executions/{run_id}/kill` go through the existing `api` client wrapper with list/detail query invalidation on success (no optimistic updates — run state is server-authoritative).
- **Live transport:** the live view opens `new EventSource('/api/executions/{runId}/events')` over the same-origin rewrite (cookie auth; the SSE route is `Depends(get_current_user)`-gated server-side, mirroring `routers/explore.py`). EventSource cannot set headers — cookie auth is the only auth, exactly as Phase 4. Each event is JSON-parsed into a `zod` `executionProgressEventSchema` in `lib/api/executions.ts` (run_id, completed, total, passed, failed, flaky, elapsed_s, status, and the per-test delta: flow_id/test id, status, attempt, duration_ms) — counters take the latest event's **absolute** values; the test list upserts the per-test row by id.
- **No fabricated status ever crosses the boundary:** per-test verdicts and run status render strictly from the server's events/payload — green/amber/red appears only when the server reports it (the Phase-6 honesty rule carried). A killed run shows a real draining state from a real flag, never a synthetic instant transition.
- Endpoints are `Depends(get_current_user)`-gated; an unauthenticated request returns 401 and the client wrapper redirects to `/login`. Artifact routes additionally apply the path-containment guard (07-RESEARCH Security Domain).

### Accessibility

- **Tables:** real `<table>` semantics via shadcn `table` (`<th scope="col">`). Run-status, run-results, and per-test verdict badges are NEVER color-only — each carries its WORD (status / verdict) alongside the colored dot/icon (WCAG 1.4.1); the dot/icon is `aria-hidden`. **Flaky vs Failed is conveyed by the WORDS "Flaky"/"Failed" AND distinct icons**, not by amber-vs-red alone.
- **Live counters:** each tile's value has an `aria-label` combining label + value (e.g. `aria-label="Failed: 2"`). The progress affordance is a real `role="progressbar"` with `aria-valuenow`/`aria-valuemin`/`aria-valuemax` (or its text "{completed} of {total} tests" as the accessible name) — never a decorative-only bar.
- **Per-test list:** the live list container is an ARIA live region — `role="log"` `aria-live="polite"` `aria-relevant="additions text"` — so newly-resolved test verdicts are announced without stealing focus (polite, not assertive — a fast suite must not flood).
- **Live pill:** `role="status"` `aria-live="polite"` so connection/run-state changes ("Running" → "Stopping…" → "Complete") are announced — the draining state is announced honestly.
- **Kill control:** a real `<button>`; the confirmation is a shadcn `dialog` with focus trapped and "Keep running" as the default focus. While draining, the button is natively `disabled` with its "Stopping…" label as the accessible state.
- **Artifact links:** real `<a>`/`<Link>` with descriptive accessible names ("Trace for {test name}"); the absent-video state is real text ("Video captured on failure only."), not an empty or dead link.
- **Charts (Recharts):** each chart `Card` has a heading (caption title) and an accessible summary — provide an `aria-label` / visually-hidden text describing the trend (e.g. "Pass rate over the last {n} runs") and ensure the chart is supplementary to the numeric data already in the runs table (the table is the source of truth; the chart is not the only way to read the data — WCAG 1.4.1 / non-text).
- **Empty/error regions:** rendered as text with a reachable action link/button (Go to scenarios / Retry / Reload); never icon-only or color-only.
- **Motion:** the live "Running" pulse and any chart entrance animation respect `prefers-reduced-motion` (drop to instant when set), mirroring the Phase-4 motion rule.

### Interaction defaults (inherited from 01-UI-SPEC, applied here)

- Focus rings: 2px accent ring on all interactive elements (tier picker, Start run, history row links, Kill, Jump to latest, artifact links, Retry/Reload).
- Sentence case throughout; verbs-first on buttons ("Start run", "Kill run", "Retry", "Reload").
- Errors render inline in the content region, never as toasts; success toasts (sonner, bottom-right) only for "Run started" and "Stopping the run".
- The live view does NOT poll for status in parallel with the stream during a live run (the SSE stream is the source of truth); it MAY `GET /api/executions/{run_id}` once on mount to resolve a 404 and to render a terminal run directly when opened after it finished (the Phase-4 contract).
